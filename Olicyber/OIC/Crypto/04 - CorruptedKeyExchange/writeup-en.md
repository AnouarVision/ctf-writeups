## Description

> Bob and Alice don't know I installed a virus on the servers they use to communicate. You have full control over the parameters used to perform the key exchange.

The server source code is provided. The server reads the DH parameters $(g, p)$ from a file that the attacker can freely overwrite, runs a key exchange between Alice and Bob, and encrypts the flag with AES-ECB using the derived shared secret.

---

## Code analysis

The server performs a Diffie–Hellman exchange between Alice and Bob using the parameters read from the file:

```python
Alice_public = pow(g, Alice_private, p)
Bob_public   = pow(g, Bob_private,   p)

Shared_secret = pow(g, Alice_private * Bob_private, p)
key = (Shared_secret % (2**(8*16) - 1)).to_bytes(16, 'big')

cipher = AES.new(key, AES.MODE_ECB)
ciphertext = cipher.encrypt(pad(FLAG, 16))
```

The vulnerability is that the parameters $(g, p)$ come entirely from attacker-controlled input, with no validation.

---

## Solution

### Step 1 — Attack: Parameter injection with g = 1

We exploit the basic algebraic property of exponentiation:

$$1^x = 1, \quad \forall x \in \mathbb{Z}$$

If the attacker forces `g = 1`, then regardless of the private keys chosen by Alice and Bob:

$$A = g^{a} \bmod p = 1^{a} \bmod p = 1$$
$$B = g^{b} \bmod p = 1^{b} \bmod p = 1$$

The shared secret becomes:

$$S = g^{ab} \bmod p = 1^{ab} \bmod p = 1$$

The AES key is therefore completely deterministic:

$$K = \left(1 \bmod (2^{128} - 1)\right)\text{.to_bytes}(16) = \underbrace{00\cdots01}_{16 \text{ byte}}$$

i.e. `0x00000000000000000000000000000001`.

---

### Step 2 — Choosing p

The file expects two hexadecimal values: `g` and `p`. For `p` one can pick a well-known MODP prime (for example the 1536-bit MODP group from RFC 3526). The server accepts it without errors.

---

### Step 3 — Decryption

With the known key `K`, AES-ECB decryption is trivial:

$$\mathrm{FLAG} = D_K(C) = \mathrm{AES\text{-}ECB\text{-}Decrypt}(K, C)$$

```python
from Crypto.Cipher import AES
from Crypto.Util.Padding import unpad
from base64 import b64decode

key = (1 % (2**(8*16) - 1)).to_bytes(16, 'big')
ct  = b64decode("HZJNGZczSDtWjBLe4zYaVi5U5CmD2lhwBpPRu4e3PLw=")

cipher = AES.new(key, AES.MODE_ECB)
print(unpad(cipher.decrypt(ct), 16).decode())
```

## Conclusions

> The security of Diffie–Hellman does not rely only on the algorithm, but also on correct parameters. Unvalidated parameters completely void cryptographic guarantees.

Three key takeaways:

1. **g = 1 collapses the group to a single element:** the cyclic subgroup generated by `g = 1` has order 1 and contains only the identity. The space of possible shared secrets collapses from $p-1$ values to exactly **1 value**, making the secret fully predictable without knowing private keys.

2. **Parameter validation is mandatory:** secure DH parameters must satisfy strict conditions — `p` should be a safe prime, `g` should generate a large-order subgroup, and `g` must not be in `{0, 1, p-1}`. Standards like RFC 7919 provide named, pre-approved DH groups to avoid naive implementations accepting degenerate parameters. The absence of any checks on `g` makes this server trivially exploitable.

3. **AES-ECB does not hide patterns:** even with a secure key, using AES-ECB is problematic — identical plaintext blocks map to identical ciphertext blocks. For short messages like a flag this is less visible, but for structured messages ECB reveals plaintext structure. Use CBC/CTR/GCM with a proper IV/nonce instead.
