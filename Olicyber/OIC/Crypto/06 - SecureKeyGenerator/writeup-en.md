# Secure Key Generator

**Competition:** OliCyber <br>
**Category:** Crypto

---

## Description

> We found an encrypted file on a remote server along with a key generator program.

The provided files are `key_generator.py` (the key generator) and `flag.enc` (the encrypted file).

---

## Code analysis

Excerpt from the key generator:

```python
def generate_secure_key():
    ts = int(datetime.timestamp(datetime.now()))
    h = sha256(int_to_bytes(ts)).digest()

    seed = int_from_bytes(h[32:])
    key = h[:32]

    random.seed(seed)
    for _ in range(32):
        key += bytes([random.randint(0, 255)])

    return key
```

The key is built by concatenating two parts:

- Deterministic part (32 bytes): `h[:32]` — SHA-256 of the Unix timestamp
- Pseudo-random part (32 bytes): 32 bytes generated by Python's `random` seeded with `int_from_bytes(h[32:])`

Total key length: 64 bytes.

---

## Solution

### Vulnerability 1 — the seed is always zero

SHA-256 produces exactly 32 bytes, so `h[32:]` is an empty sequence. Converting an empty byte sequence to int yields `0`. Therefore `random.seed(0)` is always used and the pseudo-random tail is constant across runs — no added entropy.

### Vulnerability 2 — entropy limited to the timestamp

The only remaining entropy is the Unix timestamp used as input to SHA-256. The code includes a hint comment:

```python
# 2021-03-21 17:37:40
```

Interpreting that time as a reference reduces the search space to a practical window (e.g., ±1 day → ~172k candidates), making brute force feasible.

### Cipher mode identification

`flag.enc` size (35116 bytes) is not a multiple of 16 → likely AES in a stream-like mode such as OFB. In the generator the AES key is `K[:32]` and IV is `K[32:48]`.

### Attack: brute-force the timestamp

For each candidate timestamp t in a reasonable window around the hint:

1. Reconstruct `K = GenerateKey(t)`
2. Initialize AES-OFB with `key=K[:32]` and `iv=K[32:48]`
3. Decrypt the first 48 bytes and check for a file magic (e.g., `%PDF`)

The correct timestamp found in this challenge is `1616264040` (2021-03-21 16:37:40 UTC), which is one hour earlier than the commented time (the comment used UTC+1).

## Exploit

```python
from hashlib import sha256
from datetime import datetime
import random
from Crypto.Cipher import AES

def int_to_bytes(x):
    if x == 0:
        return b''
    return x.to_bytes((x.bit_length() + 7) // 8, 'big')

def int_from_bytes(xbytes):
    return int.from_bytes(xbytes, 'big')

def generate_key_for_ts(ts):
    h = sha256(int_to_bytes(ts)).digest()
    seed = int_from_bytes(h[32:])
    key = h[:32]
    random.seed(seed)
    for _ in range(32):
        key += bytes([random.randint(0, 255)])
    return key

ciphertext = open('flag.enc', 'rb').read()
hint_ts = int(datetime(2021, 3, 21, 17, 37, 40).timestamp())

for delta in range(-86400, 86400):
    ts = hint_ts + delta
    key = generate_key_for_ts(ts)
    cipher = AES.new(key[:32], AES.MODE_OFB, key[32:48])
    pt = cipher.decrypt(ciphertext[:48])
    if pt[:4] == b'%PDF':
        print(f"Timestamp found: {ts} (delta={delta}s)")
        cipher2 = AES.new(key[:32], AES.MODE_OFB, key[32:48])
        open('flag_decrypted.pdf', 'wb').write(cipher2.decrypt(ciphertext))
        print("Decrypted file saved as flag_decrypted.pdf")
        break
```

## Conclusion

1. Out-of-bounds slicing (`h[32:]`) yields an empty byte string and a zero seed.
2. A timestamp in seconds is insufficient entropy for key material.
3. Comments in source code leak useful hints; use a CSPRNG like `os.urandom()` and avoid deriving multiple secrets from a single small-value input.
